¿Qué es un Requisito?

Es una condición o capacidad que debe cumplir el sistema para satisfacer un contrato, una norma o la necesidad de un usuario.

"Si no sabes qué construir, no puedes construir nada."

Tipos de Requisitos

Funcionales

Lo que el sistema DEBE HACER. (Ej: Registrar alumnos, imprimir factura).

No Funcionales

Cómo debe ser el sistema. (Ej: Seguridad, velocidad, diseño).

La Entrevista

Es el método más común. Consiste en una conversación directa con los interesados para obtener información detallada y aclarar dudas en tiempo real.

El Cuestionario

Serie de preguntas escritas que se envían a un gran número de personas. Es ideal para obtener estadísticas y opiniones generales de forma rápida.

La Observación

El analista observa "en vivo" cómo trabajan los empleados en su entorno real. Sirve para descubrir pasos que los usuarios olvidan mencionar en las entrevistas.

Revisión de Documentos

Analizar manuales, facturas, formularios y reportes actuales. Ayuda a entender las reglas de negocio y el flujo de datos ya existente.

Lluvia de Ideas (Brainstorming)

Reunión grupal donde todos proponen ideas sin censura. Es excelente para encontrar soluciones creativas y requisitos innovadores.

JAD (Diseño Conjunto)

Reuniones intensivas donde analistas, usuarios y gerentes trabajan juntos durante varios días para definir todo el sistema de una vez.

Prototipado de Requisitos

Crear pantallas falsas para que el usuario diga: "Sí, esto es lo que quiero" o "No, falta este botón".

Errores Comunes

  • Suponer que el usuario sabe lo que quiere.
  • Ignorar a los usuarios finales (los que usarán el programa).
  • No documentar las respuestas.

Priorización (MoSCoW)

Clasificar requisitos en:

  • Must have (Obligatorio)
  • Should have (Debería tener)
  • Could have (Podría tener)
  • Won't have (No tendrá ahora)

Documento SRS

El resultado final es el Software Requirements Specification. Es el documento legal que guía todo el desarrollo del software.
